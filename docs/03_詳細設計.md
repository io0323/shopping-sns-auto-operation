## 0. 精査による主要変更点

| # | 変更 | 理由 |
|---|------|------|
| 1 | product_metrics(日次履歴)テーブル新設 | 元設計のproductsは最新値のみで、順位変動・レビュー増加率(F-002の中核)が計算不能だった |
| 2 | contentsに status / content各フィールド / prompt_version を追加 | 承認フロー(F-005)と学習(F-006)が元スキーマでは実現不能だった |
| 3 | resultsをcontent_id紐付け+レポート期間付きに変更 | 商品単位だけでは「どの投稿文が効いたか」が分析できない |
| 4 | jobs / llm_usage / prompt_versions / import_errors を新設 | ハーネスの状態管理・コスト監視・プロンプト版管理に必須 |
| 5 | 生成系APIを非同期(202+job_id)に変更 | LLM生成は同期レスポンスに収まらない |
| 6 | 全APIにエラーレスポンス仕様・ページングを定義 | 元設計に欠落 |

## 3. バッチ・パイプライン詳細

### daily_pipeline(05:00, APScheduler)

```
1. research   : ジャンルごとにランキングAPI(上位30件)+商品検索API取得
                - リクエスト間 sleep(1.1s)。429時はバックオフ
                - products upsert, product_metrics insert(冪等: 同日再実行は上書き)
2. selection  : 直近7日メトリクスからスコア算出 → 上位10件をcandidates登録
                - 除外: 直近30日投稿済み / excluded=true / メトリクス2日分未満
3. generate   : 候補ごとにGenerator実行(並列度2)
4. evaluate   : ルールチェック → LLM評価 → <80点なら改善指示付き再生成(最大3回)
5. save       : status=evaluated(3回失敗はneeds_review)で保存
6. notify     : 「本日の候補N件(要確認M件)」を通知
※ Cost Guard: 各LLM呼出後に月間累計を確認、上限超過で3-4を停止
```

### スコア計算式(Selection、初期重み)

```
score = 0.25·rank_trend + 0.20·review_growth + 0.15·rating
      + 0.15·seasonality + 0.15·price_fit − 0.10·competition
(各要素は0〜1に正規化。重みは config/scoring.yaml で管理、合計1.0を検証)
rank_trend    = clip((rank_7d_ago − rank_today) / 30, 0, 1)
review_growth = clip(Δreview_count_7d / 20, 0, 1)
rating        = clip((review_average − 3.5) / 1.5, 0, 1)
seasonality   = config/seasonality.yaml のジャンル×月係数
price_fit     = 目標価格帯(既定1,000〜10,000円)への台形メンバーシップ
competition   = clip(review_count / 10000, 0, 1)
```

## 4. AIプロンプト設計

### 4.1 Generator Prompt(gen-v1)

```
あなたは楽天ROOMで商品を紹介するコンテンツライターです。
以下の商品情報から投稿コンテンツをJSONで生成してください。

# 制約
- description: 80〜150文字。利用シーンを1つ具体的に描く。一人称の体験談を捏造しない
- title: 30文字以内
- hashtags: 5〜8個。商品カテゴリ+利用シーン+「#楽天ROOM」を含む
- x_post: 120文字以内(URL除く)。末尾に #ad を含める
- cta: 20文字以内
- 禁止: 価格・ポイント倍率・在庫の記載 / 「最安」「No.1」「絶対」「必ず」等の断定 /
  医薬品的効能(「治る」「痩せる」等)/ 誇大表現
- 文体: 自然で押し付けがましくない。絵文字は2個まで

# 商品情報
{product_json}

# 出力(JSONのみ、コードフェンス不要)
{"title": "", "description": "", "hashtags": [], "x_post": "", "cta": ""}
```

### 4.2 Evaluator Prompt(eval-v1)

```
あなたは楽天ROOM投稿の品質審査員です。以下のコンテンツを5軸で採点してください。

# 採点軸(各0〜20点、合計100点)
1. 自然さ: AIっぽさ・不自然な言い回しがないか
2. 可読性: 長さ・改行・語彙の平易さ
3. 訴求力: 利用シーンの具体性、読者が自分ごと化できるか
4. 独自性: 提示された過去投稿({recent_posts})と表現が被っていないか
5. 規約適合: 誇大表現・断定・価格記載・効能標榜がないか(1つでもあれば0点)

# 出力(JSONのみ)
{"total": 0, "scores": {"natural":0,"readability":0,"appeal":0,"uniqueness":0,"compliance":0},
 "verdict": "pass|fail", "improvement": "failの場合、Generatorへの具体的な改善指示を1〜3点"}

# 対象コンテンツ
{content_json}
```

### 4.3 再生成ループ制御(コード側)

```
for attempt in range(3):
    content = generator.run(product, improvement_hint)
    rule_violations = rule_check(content)      # 禁止表現辞書・文字数・#ad有無
    if rule_violations: improvement_hint = rule_violations; continue
    eval = evaluator.run(content, recent_posts)
    if eval.total >= 80: return save(content, status="evaluated")
    improvement_hint = eval.improvement
return save(content, status="needs_review")
```

### 4.4 禁止表現辞書(ルールベース、抜粋)

```
断定・誇大: 最安, 最強, No.1, 日本一, 絶対, 必ず, 100%, 永久
薬機法系 : 治る, 完治, 痩せる, 効く, アンチエイジング, 医者いらず
価格系   : 円(数字直後), ポイント◯倍, セール中, 在庫
※ config/ng_words.yaml で管理。正規表現対応
```

## 5. ログ設計

| ログ | 内容 | 形式 |
|------|------|------|
| API実行ログ | 楽天API: URL(キー除去)・ステータス・所要時間 | JSON Lines |
| AI生成ログ | プロンプト版・入出力(本文はDB、ログはID参照)・トークン数 | DB(llm_usage)+JSONL |
| 評価ログ | スコア内訳・verdict・再生成回数 | DB |
| 操作ログ | 承認/除外/編集/投稿マーク(操作・対象ID・時刻) | DB |
| エラーログ | スタックトレース・ジョブID | JSONL |
| 保持期間 | JSONL 90日ローテーション(logrotate相当を内蔵)、DBは無期限 | |
