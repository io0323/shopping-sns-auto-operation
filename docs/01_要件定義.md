# 要件定義書 — Shopping SNS Auto Operation

改訂版 v1.1(精査反映)

---

## 0. 精査による主要変更点

| # | 変更 | 理由 |
|---|------|------|
| 1 | 「自動投稿」を要件から除外し「半自動(下書き生成+人間が投稿)」に統一 | 楽天ROOMに投稿用公式APIが存在しない。UI自動化・スクレイピングによる投稿は楽天ROOM利用規約違反となり、アカウント停止・アフィリエイト報酬没収のリスクがある |
| 2 | 成果データ手動インポート機能(F-007)を新設 | ROOMのクリック・成果データは楽天アフィリエイトレポートにしかなく、API非公開。CSVインポートが唯一の合法的な取得手段 |
| 3 | 非機能要件を単一ユーザー・ローカル運用前提に現実化 | 稼働率99%/同時100件/JWTは個人ツールには過剰。代わりにコスト上限・レート制限遵守を追加 |
| 4 | AI生成の性能要件を「同期10秒以内」から「非同期ジョブ」に変更 | LLM生成+品質評価+再生成ループは10秒に収まらない。バッチ/非同期前提が現実的 |
| 5 | 投稿文への価格・ポイント倍率の記載を原則禁止 | 価格は変動するため、投稿後に実価格と乖離すると景品表示法・アフィリエイト規約上のリスク |
| 6 | 開発をPhase 1〜3に分割 | 学習機能(F-006)は実績データが数十件以上蓄積されるまで機能しない。最初から作っても検証不能 |

## 1. システム概要

### 1.1 システム名

Shopping SNS Auto Operation(略称: SSAO)

### 1.2 開発目的

- 商品選定時間の削減(現状: 手動リサーチ → AI候補抽出)
- 投稿コンテンツ作成工数の削減(現状: 手作業 → AI生成+品質自動評価)
- データに基づく運用改善(成果データの構造化蓄積と分析)
- 成功パターンのプロンプトへのフィードバックによる継続的最適化
- 長時間自律実行可能なAIハーネス構成(ただし**投稿操作は常に人間**)

### 1.3 スコープ定義(重要)

**自動化する範囲**

- 楽天APIによる商品情報の定期取得・履歴蓄積
- 商品スコアリングと候補抽出
- 投稿文・タイトル・ハッシュタグ・X投稿文の生成
- 生成物の品質評価と再生成
- 成果データ(インポート済み)の分析とプロンプト改善提案

**自動化しない範囲(規約上の理由により恒久的に対象外)**

- 楽天ROOMへの投稿操作(コレ!追加、オリジナル写真投稿、コメント)
- 楽天ROOM・楽天アフィリエイト管理画面からのデータのスクレイピング
- X(Twitter)への自動投稿(Phase 2でX API有料プラン契約時のみ検討可。無契約での自動化は行わない)

## 2. 利用者・前提条件

| 項目 | 内容 |
|------|------|
| 主利用者 | 楽天ROOM運用者本人(単一ユーザー) |
| 実行環境 | ローカルPC または 自宅サーバー(常時公開しない) |
| 前提 | 楽天アカウント / 楽天ROOM利用 / 楽天ウェブサービス(アプリID)登録済み / 楽天アフィリエイトID保有 / LLM APIキー(Anthropic等)保有 |

## 3. 業務要件

### 3.1 現状課題

- 売れる商品の発見に時間がかかる(ランキング・レビューの手動巡回)
- 投稿文作成が完全手作業で、1件あたりの作成時間が長い
- 成功パターンが個人の記憶頼みで蓄積されない
- 分析が感覚的(どのジャンル・文面が成果に繋がったか不明)

### 3.2 導入後の業務フロー

```
[毎朝 自動]  商品取得 → 分析・スコアリング → コンテンツ生成 → 品質評価 → 候補保存
[毎朝 手動]  候補レビュー(承認/修正/除外) → 投稿文コピー → ROOMアプリで投稿(1件あたり数十秒)
[週次 手動]  楽天アフィリエイトレポートCSVをダウンロード → システムにインポート
[週次 自動]  実績分析 → 成功パターン抽出 → プロンプト改善提案
```

導入後の人間の作業は「承認判断」「投稿操作」「CSVダウンロード」の3つに集約される。

## 4. 機能要件

### F-001 商品情報取得機能

| 項目 | 内容 |
|------|------|
| 概要 | 楽天ウェブサービス(楽天市場商品検索API・楽天市場ランキングAPI・ジャンル検索API)から商品情報を取得し、**日次スナップショットとして履歴保存**する |
| 取得項目 | 商品名 / 商品コード(itemCode) / ジャンルID / 価格 / レビュー件数 / レビュー平均 / 商品URL(アフィリエイトURL) / 商品画像URL / ショップ名・ショップコード / ランキング順位 / ポイント倍率 / 取得日時 |
| 入力 | 監視対象ジャンルIDリスト(設定ファイルで管理) |
| 出力 | 商品マスタ + 日次メトリクス履歴 |
| 制約 | 楽天APIレート制限(1リクエスト/秒目安)を遵守。リクエスト間に1秒以上のウェイトを入れる |

> 精査メモ: 元設計は「入力: なし」だったが、対象ジャンルの指定は必須。また順位変動・レビュー増加率(F-002)を計算するには単発取得でなく履歴が必要なため、スナップショット保存を要件化。

### F-002 商品分析機能

| 項目 | 内容 |
|------|------|
| 概要 | 蓄積された履歴から商品評価スコアを算出し、投稿候補を抽出 |
| 評価軸 | 順位変動(直近N日) / レビュー増加率 / レビュー平均 / 季節性(ジャンル×月の重み) / 価格帯適合(自ROOMフォロワー層に合う価格帯) / 競合度(ROOM内投稿数の代理指標としてレビュー件数規模を使用) |
| 出力 | スコア付き商品候補一覧(スコア内訳付き) |
| 制約 | スコアの重みは設定ファイルで変更可能とし、Learning Agentの改善提案の適用先とする |

### F-003 AIコンテンツ生成機能

| 項目 | 内容 |
|------|------|
| 生成対象 | ROOM説明文(コレ!コメント) / タイトル / ハッシュタグ(推奨5〜8個) / X投稿文(URL分を除き実質120字以内) / CTA |
| 入力 | 商品情報 + 商品レビュー傾向 + 過去の高成果投稿パターン(Phase 3以降) |
| 制約 | 価格・ポイント倍率・在庫状況の断定記載禁止(変動リスク) / 誇大表現・優良誤認表現禁止 / 医薬品的効能効果の標榜禁止(健康食品・コスメ) / ステマ規制対応として広告であることが分かる表記(#ad 等)を含める |

### F-004 AI品質評価機能

| 項目 | 内容 |
|------|------|
| 評価軸(100点満点) | 自然さ / 可読性 / 訴求力 / 過去投稿との重複率 / 規約適合性(誇大表現・禁止表現・価格記載の有無) |
| 判定 | 80点以上: 候補として保存 / 80点未満: 改善指示付きで再生成 |
| 制約 | **再生成は最大3回**。3回で80点に達しない場合は「要人間確認」ステータスで保存(無限ループ・コスト暴走防止) |
| 補助 | 禁止表現(「最安」「No.1」「絶対」「完治」等)はLLM評価の前に**ルールベースで機械チェック**し、確実に検出する |

### F-005 投稿候補管理機能

| 項目 | 内容 |
|------|------|
| 概要 | 生成済みコンテンツの承認・修正・除外・スケジュール管理 |
| 機能 | 候補一覧表示(スコア・品質点順) / 本文編集 / 承認・除外 / 投稿予定日時の設定 / **投稿文のワンタップコピー** / 投稿完了の手動マーク |
| 出力 | 承認済み投稿キュー(人間が投稿する順番待ちリスト) |

> 精査メモ: 元設計のPublisher Agent(自動投稿)はここに吸収。システムの出口は「コピーしやすい下書き」であり、投稿ボタンは持たない。

### F-006 学習機能(Phase 3)

| 項目 | 内容 |
|------|------|
| 概要 | インポート済み実績データと投稿コンテンツを突合し、成功パターンを抽出。生成プロンプトの改善案とスコア重みの調整案を提示 |
| 入力 | F-007でインポートされた実績 + contents + products |
| 出力 | パターン分析レポート / プロンプト改善提案(人間が承認して初めて反映) |
| 制約 | プロンプトの自動書き換えは行わない(人間承認ゲート必須)。実績30件未満の間は「データ不足」を明示しレポートのみ |

### F-007 実績データインポート機能(新設)

| 項目 | 内容 |
|------|------|
| 概要 | 楽天アフィリエイトのレポートCSV(クリック・成果)を取り込み、商品・コンテンツに紐付ける |
| 入力 | 楽天アフィリエイト管理画面から手動ダウンロードしたCSV |
| 処理 | 文字コード(Shift-JIS)変換 / itemCodeまたはURLベースの突合 / 重複取込防止(同一期間の再取込は上書き) |
| 出力 | results テーブルへの反映 |

## 5. 非機能要件

### 5.1 性能

| 項目 | 要件 |
|------|------|
| 楽天API取得バッチ | 対象ジャンル20件・上位30商品/ジャンルを30分以内(レート制限1req/秒の遵守を優先) |
| AIコンテンツ生成 | 非同期ジョブとして実行。1商品あたり生成+評価+再生成込みで平均2分以内 |
| 管理画面の画面応答 | 一覧表示 2秒以内 |
| 同時利用 | 単一ユーザー前提。同時実行制御はジョブキューの直列/少数並列で担保 |

### 5.2 コスト(新設)

| 項目 | 要件 |
|------|------|
| LLM API費用 | 月額上限を設定ファイルで定義(例: 月3,000円)。トークン使用量を全リクエストで記録し、上限到達時は生成バッチを自動停止して通知 |
| モデル選択 | 生成: 中位モデル / 評価: 軽量モデル、のように役割別にモデルを使い分け可能にする |

### 5.3 可用性

- ローカル/自宅サーバー運用のベストエフォート。バッチ失敗時は次回実行でリカバリできる冪等設計とする
- 稼働率のSLAは設定しない(投稿は人間が行うため、システム停止=機会損失であって事故ではない)

### 5.4 セキュリティ

- APIキー(楽天アプリID・アフィリエイトID・LLMキー)は `.env` 管理とし、リポジトリにコミットしない(`.gitignore` 必須。公開リポジトリ化する場合は特に注意)
- 管理画面はlocalhostバインドを既定とする。外部公開する場合のみBasic認証+HTTPS(Caddy/Cloudflare Tunnel等)
- 操作ログ(承認・除外・編集)を保存

> 精査メモ: JWTは単一ユーザー・非公開運用では不要と判断し削除。公開時の代替手段を明記。

### 5.5 保守性

- エージェント単位でモジュール分離、API単位で疎結合
- プロンプトはコードから分離し、バージョン管理されたテキストファイル(またはDB)で管理
- 主要ロジック(スコア計算・禁止表現チェック・CSV突合)は単体テストを必須とする

### 5.6 コンプライアンス(新設)

- 楽天ウェブサービス利用規約・楽天アフィリエイト規約・楽天ROOM利用ガイドラインの遵守
- 生成コンテンツはステマ規制(景品表示法)対応の広告明示を含む
- 商品画像の利用は楽天APIが返すURL参照のみとし、加工・再配布しない

## 6. 開発フェーズ

| Phase | 範囲 | 完了条件 |
|-------|------|----------|
| Phase 1 | F-001 / F-002 / F-003 / F-004 + CLI or 最小UI | 毎朝、スコア順の投稿候補(品質評価済み)が自動で用意される |
| Phase 2 | F-005(承認UI・コピー・スケジュール) / F-007(CSVインポート) | 承認→コピー→投稿→実績取込の運用サイクルが回る |
| Phase 3 | F-006(学習・改善提案) / (任意)X API連携 | 実績に基づく改善提案が週次で出力される |
