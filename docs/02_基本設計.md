# 基本設計書 — Shopping SNS Auto Operation

改訂版 v1.1(精査反映)

---

## 0. 精査による主要変更点

| # | 変更 | 理由 |
|---|------|------|
| 1 | Publisher Agent → **Export Agent** に再定義 | 楽天ROOMに投稿APIが無く、自動投稿は規約違反リスク。出口は「下書きエクスポート+コピー支援」 |
| 2 | Redisを初期構成から削除、ジョブはAPScheduler+DBキューで代替 | 単一ユーザーの日次バッチにRedis/Celeryは過剰。Phase 3で並列度が必要になった時点で導入判断 |
| 3 | PostgreSQL → **SQLiteで開始し、必要時にPostgreSQLへ移行可能な構成**(SQLAlchemy経由) | ローカル単一ユーザーではSQLiteで十分。運用コストゼロ |
| 4 | Harness Engineの責務を明文化(状態管理・リトライ・承認ゲート・コスト監視) | 元設計では箱の名前のみで責務が未定義だった |
| 5 | Import Agent(実績CSV取込)を追加 | F-007新設に対応 |

## 1. システム構成

```
┌──────────────────────────────────────────────┐
│ Frontend: Next.js (App Router)               │
│  - 候補レビューUI / ダッシュボード / CSV取込  │
└──────────────┬───────────────────────────────┘
               │ REST (localhost)
┌──────────────▼───────────────────────────────┐
│ Backend: FastAPI                             │
│  ├ REST API層                                │
│  ├ Harness Engine(オーケストレータ)         │
│  │   ├ Pipeline Runner(日次/週次パイプライン)│
│  │   ├ Job Queue(DBテーブルベース)          │
│  │   ├ Retry / Checkpoint / Cost Guard       │
│  │   └ Human-in-the-loop Gate(承認待ち制御) │
│  ├ Agents                                    │
│  │   ├ Planner Agent                         │
│  │   ├ Research Agent                        │
│  │   ├ Selection Agent                       │
│  │   ├ Generator Agent                       │
│  │   ├ Evaluator Agent                       │
│  │   ├ Export Agent(旧Publisher)            │
│  │   ├ Import Agent(新設)                   │
│  │   └ Learning Agent                        │
│  └ Scheduler: APScheduler                    │
└──────┬───────────────┬───────────────────────┘
       │               │
┌──────▼──────┐  ┌─────▼─────────────────────┐
│ SQLite      │  │ 外部API                    │
│ (SQLAlchemy)│  │  - 楽天ウェブサービス       │
│             │  │  - LLM API (Anthropic)     │
└─────────────┘  └───────────────────────────┘
```

技術選定の根拠:

- **FastAPI + SQLAlchemy**: 非同期対応、Pydanticによる型安全なAPI。SQLite→PostgreSQL移行がURL変更のみで可能
- **APScheduler**: プロセス内スケジューラで日次05:00バッチを実現。cron依存なし
- **Next.js**: 候補レビューUIはテーブル+編集+コピー中心。shadcn/ui相当のシンプル構成
- **LLM SDK**: Anthropic SDK。モデルは役割別に設定ファイルで切替(生成=中位、評価=軽量)

## 2. Harness Engine 設計

### 2.1 責務

| 機能 | 内容 |
|------|------|
| Pipeline定義 | エージェントの実行順序をパイプラインとして宣言的に定義(daily_pipeline / weekly_pipeline) |
| 状態管理 | 各ステップの状態(pending / running / done / failed / waiting_approval)をDBに永続化。途中失敗時は失敗ステップから再開 |
| リトライ | 外部API失敗時は指数バックオフで最大3回。楽天API 429時は待機時間を延長 |
| Cost Guard | LLM呼び出しごとにトークン数・概算費用を記録。月次上限超過でパイプラインを停止し通知 |
| 承認ゲート | Export以降のステップは人間の承認がない限り進まない(状態 waiting_approval で停止) |
| 通知 | パイプライン完了・失敗・コスト警告をログ+(任意)メール/Slack Webhookで通知 |

### 2.2 パイプライン定義

```
daily_pipeline (05:00):
  Research → Selection → Generator ⇄ Evaluator → 候補保存 → [承認ゲート] → Export

weekly_pipeline (日曜 06:00, Phase 3):
  Import済み実績チェック → Learning → 改善提案保存 → [承認ゲート] → 設定反映
```

## 3. エージェント基本設計

### Planner Agent(Phase 3で本格化、Phase 1は設定ファイル代替)

| 項目 | 内容 |
|------|------|
| 役割 | 月間目標と実績から投稿戦略(ジャンル配分・日次投稿数・注力価格帯)を策定 |
| 入力 | 月間目標(クリック数/成果件数)、直近実績、季節イベントカレンダー |
| 出力 | 投稿戦略(strategyテーブル)。Research/Selectionのパラメータとして参照される |
| 実装方針 | Phase 1では `config/strategy.yaml` を人間が書く。Phase 3でLLM提案+人間承認に昇格 |

### Research Agent

| 項目 | 内容 |
|------|------|
| 役割 | 楽天APIからの商品情報収集と履歴保存 |
| 入力 | 戦略で指定されたジャンルIDリスト |
| 処理 | ランキングAPI(ジャンル別上位) + 商品検索API(レビュー件数順)を取得。1req/秒遵守。商品マスタをupsertし、日次メトリクスをinsert |
| 出力 | products / product_metrics 更新 |
| 備考 | LLMを使わない純ロジックエージェント |

### Selection Agent

| 項目 | 内容 |
|------|------|
| 役割 | スコアリングと候補選定 |
| 入力 | 直近メトリクス履歴付き商品一覧 |
| 処理 | 重み付きスコア = w1·順位変動 + w2·レビュー増加率 + w3·評価平均 + w4·季節性 + w5·価格帯適合 − w6·競合度。重みは設定管理 |
| 出力 | スコア上位N件(既定10件/日)を候補としてマーク。**スコア内訳を保存**(後の学習と説明可能性のため) |
| 備考 | LLM不使用。除外条件: 直近30日に投稿済みの同一商品 / 除外ジャンル / アダルト等 |

### Generator Agent

| 項目 | 内容 |
|------|------|
| 役割 | 投稿コンテンツ一式の生成 |
| 入力 | 商品情報 + スコア内訳 + 生成プロンプト(バージョン管理) |
| 出力 | title / description / hashtags / x_post / cta を含むcontentsレコード(status=draft) |
| 制約 | 価格・ポイント断定記載禁止、広告明示タグ必須、文字数制約はプロンプトで指定しコード側でも検証 |

### Evaluator Agent

| 項目 | 内容 |
|------|------|
| 役割 | 品質判定と再生成指示 |
| 処理 | ①ルールベースチェック(禁止表現辞書・文字数・必須タグ)→ ②LLM評価(自然さ/可読性/訴求力/重複率/規約適合の5軸100点)→ ③80点未満なら改善指示を添えてGeneratorへ差戻し(最大3回) |
| 出力 | quality_score + 軸別内訳 + 判定コメント。3回失敗時は status=needs_review |
| 備考 | 重複率は過去自コンテンツとのembedding類似度 or n-gram一致で機械算出し、LLM評価と併用 |

### Export Agent(旧 Publisher Agent)

| 項目 | 内容 |
|------|------|
| 役割 | 承認済みコンテンツの出力支援。**投稿は行わない** |
| 入力 | status=approved のコンテンツ |
| 出力 | 投稿用テキスト(ROOM用/X用)の整形、ワンタップコピー用データ、投稿チェックリスト、投稿予定順のキュー |
| 将来拡張 | X API(有料プラン)契約時のみ、X自動投稿をオプション実装。楽天ROOMへの自動化は恒久対象外 |

### Import Agent(新設)

| 項目 | 内容 |
|------|------|
| 役割 | 楽天アフィリエイトレポートCSVの取込 |
| 処理 | Shift-JIS→UTF-8変換 / カラムマッピング / itemCode・URL突合 / 期間重複時は上書き / 突合不能行はレポート出力 |
| 出力 | results 更新 + 取込サマリ |

### Learning Agent(Phase 3)

| 項目 | 内容 |
|------|------|
| 役割 | 実績とコンテンツ特徴の相関分析、改善提案 |
| 処理 | 高成果/低成果コンテンツの特徴比較(ジャンル・文体・ハッシュタグ・投稿時間帯)→ パターン抽出 → プロンプト改善案とスコア重み調整案を生成 |
| 出力 | 週次レポート + 改善提案(status=proposed)。**人間承認後にのみ** prompt_versions / スコア重み設定へ反映 |
| ガード | 実績30件未満は分析をスキップし「データ蓄積中」レポートのみ |

## 4. データフロー概要

```
[05:00] Research ─→ products / product_metrics
             │
        Selection ─→ candidates(スコア内訳付き)
             │
        Generator ─→ contents(draft)
             │  ↑改善指示(最大3回)
        Evaluator ─→ contents(evaluated / needs_review)
             │
        ── 承認ゲート(人間: UI上で approve/edit/reject)──
             │
        Export ─→ 投稿キュー表示・コピー → 人間がROOMアプリで投稿 → 投稿済みマーク
             
[週次]  人間: アフィリエイトCSV DL → Import ─→ results
        Learning ─→ 改善提案 → 人間承認 → プロンプト/重み更新
```

## 5. エラーハンドリング方針

| 事象 | 対応 |
|------|------|
| 楽天API 429/5xx | 指数バックオフ(1s→4s→16s)、3回失敗でそのジャンルをスキップし継続 |
| LLM APIエラー | 同上。パイプラインは商品単位で独立させ、1件の失敗が全体を止めない |
| コスト上限到達 | Generatorステップを即停止。処理済み分は保持。通知 |
| CSV突合失敗 | 失敗行をimport_errorsに保存しUI表示。全体は継続 |
| バッチ二重起動 | パイプラインIDによる排他制御(DBロック) |
